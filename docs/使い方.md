# BreezeFlow 使い方ガイド

## 起動方法

1. Visual Studio 2022 で `BreezeFlow.csproj` を開く
2. NuGet パッケージの復元を待つ（初回は数分かかる場合あり）
3. **F5** キーでビルド＆起動（または Ctrl+F5）

---

## 基本的な操作フロー

```
① DB接続登録  →  ② ステップ追加  →  ③ 設定入力  →  ④ ステップ接続  →  ⑤ 実行
```

---

## ① DB 接続設定の登録

メニュー **[ツール] → [DB 接続設定]** を開き、利用したい DB の接続を登録します。  
登録した接続は各 Input / Output ステップのダイアログで共通して選択できます。

### 対応 DB と既定値

| DB 種別 | アイコン | 既定ポート | 主な入力項目 |
|---|---|---|---|
| Oracle | 🔶 | 1521 | サーバー / ポート / サービス名 / ユーザー / パスワード |
| MySQL | 🐬 | 3306 | サーバー / ポート / DB名 / ユーザー / パスワード |
| PostgreSQL | 🐘 | 5432 | サーバー / ポート / DB名 / ユーザー / パスワード |
| SQL Server | 🟦 | 1433 | サーバー / ポート / DB名 / Windows認証 or ユーザー/パスワード |
| MariaDB | 🦭 | 3306 | サーバー / ポート / DB名 / ユーザー / パスワード（MySQL と互換） |
| SQLite | 🪶 | (なし) | データベースファイル（[参照…] で選択／ファイルが無ければ自動生成）|

各接続にはアイコン付きの表示名（例: `🐘 [PRD] 本番PostgreSQL 🔒`）が一覧に表示されます。  
**[🔗 接続テスト]** ボタンで疎通確認ができます。  
接続設定は `connections.json`（実行ファイルと同じフォルダ）に自動保存されます。

### 入力は自動で保存されます

接続マネージャには「保存」ボタンはありません。次のタイミングで自動的に `connections.json` へ書き込まれます。

| タイミング | 動作 |
|---|---|
| 一覧で**別の接続を選択** | 直前まで編集していた接続を保存してから新しい選択を表示 |
| **DB 種別**を変更 | 新しい DB の接続文字列を組み立てて即保存 |
| **環境タグ**・**🔒 Select 専用**を変更 | 即保存（Production への昇格は確認ダイアログあり）|
| **＋ 追加**を押す | 直前の接続を保存 → 新規エントリを作成 → 一覧へ追加 |
| **ダイアログを閉じる** | 編集中の最終状態を保存 |
| **🗑 削除 / ↑↓ 並び替え** | 即保存 |

> 💡 **＋ 追加の連打防止**: 現在編集中の接続が「新しい接続」のままサーバー / SQLite ファイルパスも未入力の状態だと、「先に接続名とサーバーを入力してから次の接続を追加してください」と案内し、新規追加をブロックします。空エントリが大量に積み上がるのを防ぎます。

> 💡 接続一覧の右側にある **↑/↓ ボタン** で表示順を自由に並び替えできます。各ステップダイアログの接続ドロップダウンの順序にも自動反映されます。

> 💡 SQL Server で TLS 証明書エラーが出る場合は、ビルダーで `TrustServerCertificate=true` を自動付与しているため通常は問題ありませんが、運用時はサーバ証明書の検証ポリシーをご確認ください。

> ⚠ **接続名のリネームについて**: パイプラインは接続を Guid (`ConnectionId`) で参照しているため、接続名を変えても既存パイプラインの動作には影響しません。サーバー / DB 名 / ユーザーを書き換えた場合は **同じ Guid を参照する全てのパイプライン**が新しい接続先に切り替わるため、接続マネージャ下部に表示される「現在のパイプラインで N 個のステップから参照中: ...」を必ず確認してください。

### 接続の安全機構 (誤実行を防ぐ)

DB 接続を誤って本番に向けたまま実行する事故を防ぐため、接続には次の 2 つの**安全属性**を付けられます。

#### 環境タグ (DEV / STG / PRD)

接続マネージャの「環境タグ」ドロップダウンで設定します。

| タグ | 用途 | 識別色 | 挙動 |
|---|---|---|---|
| **DEV** (Development) | 開発用 (既定) | 🔵 青 `#1565C0` | 制限なし |
| **STG** (Staging) | 検証用 | 🟠 オレンジ `#F57C00` | 実行制限はなし。識別のためキャンバスの接続ラベルが**オレンジ**で表示される (`[STG]` 付き) |
| **PRD** (Production) | 本番用 | 🔴 赤 `#D32F2F` | キャンバスのステップノードに**赤枠**、接続ラベルが**赤太字**で `[PRD]` 表示。書き込み系ステップを含むパイプラインを ▶ 実行すると、**実行前に最終確認ダイアログ**が出る |

> 接続を Production に昇格させるときは「本当に Production としてマークしますか？」と確認が出ます。

識別色は次の 3 箇所で同じ色で表示されるため、誤接続を一目で見つけられます:

- **接続マネージャの「接続一覧」**: 各エントリの左に色付きの〇
- **接続マネージャの「環境タグ」ドロップダウン**: DEV / STG / PRD 選択肢の左に色付きの〇
- **各ステップダイアログの「DB 接続」ドロップダウン**: 選択中の接続および候補リストに色付きの〇 (DBInput / DBOutput / DBUpdate / DBDelete / Insert‐Update / Exec SQL)

#### Select 専用 (🔒 書き込み禁止)

接続マネージャの「🔒 Select 専用」チェックを ON にすると、その接続を使う以下のステップは**実行時に拒否**されます:

- DB Output / DB Delete / DB Update / Insert/Update
- Exec SQL

SELECT のみの DB Input は通常どおり実行できます。本番マスタを参照だけしたい場合に有効です。

### 接続編集時の参照表示

接続マネージャをツールバーの **🔧 DB 接続設定** から開くと、選択中の接続が現在のパイプラインの**何ステップから参照されているか**が一覧表示されます。

```
⚠ 現在のパイプラインで 3 個のステップから参照中: 受注取得, 在庫更新, ログ書込
```

接続の中身（サーバー / DB 名 / ユーザー）を変更すると、これら全ステップが新しい接続先を使うようになるため、参照前にどこに影響するか把握できます。

### パイプライン実行時の事前検査

▶ 実行ボタンを押した直後に、すべての DB ステップについて以下を検査します。

| チェック | 不合格時の挙動 |
|---|---|
| `ConnectionId` がレジストリで解決できるか | ❌ **実行ブロック**。ステップノードに「⚠ 接続未解決」の赤バッジ表示 |
| Read-only 接続への書き込み | ❌ **実行ブロック** |
| Production 接続への書き込み | ⚠ **確認ダイアログ** (Yes で続行、No でキャンセル) |
| Exec SQL の SQL に DML/DDL を含むか | INSERT/UPDATE/DELETE/MERGE/DROP/TRUNCATE/ALTER/CREATE が含まれるときのみ書き込み扱い。SELECT のみなら Production 確認をスキップ |

> 旧バージョンでは `ConnectionId` が解決できないとき旧形式 `Settings["ConnectionString"]` にサイレントフォールバックしていましたが、誤った DB を叩く原因になるため**廃止**しました。解決失敗時は必ず明示エラーで停止します。

## ② ステップの追加

左側の **ステップパレット** からボタンを **ダブルクリック** すると、キャンバスに追加されます。

### Input グループ（青）

| ステップ | 説明 |
|---|---|
| 🗄 DB Input | 主要 RDBMS（Oracle / MySQL / PostgreSQL / SQL Server / MariaDB / SQLite）から SELECT でデータ取得 |
| 📗 File Input | Excel (.xlsx) ファイルからデータ読み込み |

### Output グループ（緑）

| ステップ | 説明 |
|---|---|
| 💾 DB Output | 主要 RDBMS のテーブルへ INSERT / UPSERT |
| 📄 File Output | Excel / CSV / TSV / TXT ファイルへ書き出し（日時トークン対応）|

### Transform グループ（オレンジ）

| ステップ | 説明 |
|---|---|
| 🔍 Filter | 条件に一致する行は **pass** ポート、不一致行は **fail** ポートへ振り分け |
| 🔀 Switch | フィールド値で複数の出力ポートへ動的に分岐 |
| 🧮 Calculation | 演算結果を新フィールドとして追加 |
| 📋 Select Values | フィールドを選択・リネーム（不要なフィールドを除去） |
| 📌 Set Variable | TODAY / YESTERDAY などの日付変数を生成 |

### DB Operation グループ（ピンク）

| ステップ | 説明 |
|---|---|
| 🗑 DB Delete | 入力データのキーでDB行を削除 |
| 🔄 Insert/Update | キー一致→UPDATE、なければ INSERT（UPSERT）|
| ✏️ DB Update | キー一致→UPDATE のみ（INSERT しない）|
| ⚡ Exec SQL | 任意の SQL を実行 |

> いずれも対応 6 DB（Oracle / MySQL / PostgreSQL / SQL Server / MariaDB / SQLite）で動作。UPSERT 構文は接続先 DB に応じて `MERGE` / `ON DUPLICATE KEY UPDATE` / `ON CONFLICT DO UPDATE` を自動選択します。

### Utility グループ（紫）

| ステップ | 説明 |
|---|---|
| 🔗 Merge Join | 2つの入力ストリームをキーで結合 |
| 🔍 Table Compare | 2 ストリームをキー + 列値で突き合わせ、差分を抽出（データ移行検証用）|
| ⬜ Dummy | 何もしないパススルー（分岐終端）|

#### Table Compare（データ移行検証）

2 つの入力ストリーム（移行元 / 移行先）をキーで突き合わせて、各列の値の一致／不一致を抽出するステップです。`MergeJoin` と同じく **最初に接続したストリームが Left（移行元）、2番目が Right（移行先）** になります。

**典型的なパイプライン例**

```
[DBInput: src.orders]  ──┐
                          ├─ [🔍 Table Compare]  ──  [📄 File Output: diff.xlsx]
[DBInput: dst.orders]  ──┘
```

**設定項目**

| 項目 | 説明 |
|---|---|
| キーフィールド | カンマ区切り。両側を突き合わせるキー（例: `emp_id` / `dept_id,emp_id`）|
| 比較フィールド | カンマ区切り。空欄なら自動推定（両側の全列 - キー列）|
| 両側 NULL を等しいとみなす | 既定 ON。OFF にすると `NULL` 同士も DIFF 扱い |
| 文字列の大文字小文字を無視 | "Alice" と "ALICE" を一致扱い |
| 文字列の前後空白を除去してから比較 | `"  Alice  "` と `"Alice"` を一致扱い |
| MATCH 行も出力する | 既定 OFF（差分行のみ）。 ON にすると全行出力 |

**出力スキーマ**

| 列 | 内容 |
|---|---|
| 各キー列（指定した名前のまま）| Left に値があれば Left、無ければ Right |
| 各比較列 `col_L` / `col_R` | Left / Right 側の値 |
| `_status` | `MATCH` / `DIFF` / `ONLY_LEFT` / `ONLY_RIGHT` |
| `_diff_columns` | DIFF 行のみ、差異のある比較列名のカンマ区切り |

**判定ルール**
- 数値同士は decimal で比較（`1` と `1.0` は一致）
- `DateTime` 同士はそのまま比較
- それ以外は文字列化して比較（オプションで `IgnoreCase` / `TrimStrings` 適用）

**実行ログのサマリ例**
```
TableCompare: MATCH 1234 / DIFF 7 / ONLY_LEFT 0 / ONLY_RIGHT 2  (MATCH 行は省略)
```

#### Filter（pass/fail で行を振り分け）

Filter ステップは **pass (一致行)** と **fail (不一致行)** の 2 つの出力ポートを持ちます。ノード右側に縦に並ぶ丸が出力ポートで、色で識別できます。

| ポート | 色 | 内容 |
|---|---|---|
| **pass** | 🟢 緑 | 条件に一致した行 |
| **fail** | 🟠 オレンジ | 条件に一致しなかった行 |

それぞれの丸からドラッグして別のステップへ接続できます。**fail ポートに何も接続しなければ、不一致行は捨てられます**（旧 Filter と同じ挙動）。

接続線にも `pass` / `fail` のラベルが小さく表示されるので、後から見ても「どちらの分岐か」が分かります。

#### Switch（フィールド値で複数分岐）

評価対象のフィールド値と Cases のマッピングを照らし合わせ、ヒットしたケースのポートへ行を流します。どの Case にも当てはまらない行は `default` ポートへ。

**設定項目**

| 項目 | 説明 |
|---|---|
| 評価フィールド名 | このフィールドの値を Cases と完全一致 (Ordinal) で比較 |
| Cases | 1 行 1 ケース。書式は `値|BranchKey|表示名` (BranchKey と表示名は省略可) |
| default ポートを使う | OFF にすると「どの Case にも当たらなかった行」は破棄される |

**Cases の書き方**

```
東京|tokyo
大阪|osaka|大阪支店
名古屋
```

- `東京|tokyo` → 値「東京」の行を **tokyo** ポートへ
- `大阪|osaka|大阪支店` → 値「大阪」の行を **osaka** ポートへ。ノードのツールチップには「大阪支店」と表示
- `名古屋` → BranchKey 省略時は値そのもの（名古屋）を Key にする

**評価ルール**
- 上から順に最初にヒットした Case を採用 (同一値の Case が複数あれば先勝ち)
- 比較は文字列の完全一致
- フィールドが NULL / 欠落の場合は `default` (または破棄)

OK で設定を保存するとノードの出力ポート数が自動的に増減します。Cases を追加するたびに新しい丸が現れます。

**典型例: 地域別ファイル分割**

```
                  ┌── pass(東京) ──→ [File Output: 東京.xlsx]
[DBInput] ── [🔀 Switch: Region] ── osaka  ──→ [File Output: 大阪.xlsx]
                  └── default ──→ [File Output: その他.xlsx]
```

## ③ ステップの設定

ステップノードを **ダブルクリック** または **右クリック → 設定** で設定ダイアログが開きます。

### DB Input の設定

1. ドロップダウンから **接続** を選択（全 6 DB 共通の UI）
2. **[↓ 一覧取得]** でテーブル一覧をプルダウンに読み込み（任意）
3. **SQL クエリ** を入力（または **[SQL 自動生成]** で `SELECT * FROM table` を生成）
4. **[🔗 接続テスト]** で疎通確認、**[👁 プレビュー]** で先頭 N 行を確認

> 💡 **SQL 入力欄のキー操作** — `Tab` で半角スペース 4 個を挿入、`Ctrl+Tab` で次のコントロールへフォーカス移動、`Shift+Tab` は WPF 既定挙動。DBInput / Exec SQL / SQL エディタ（ダブルクリック起動）で共通です。

> 💡 プレビューグリッドではカラム名の `_` が WPF の DataGrid ヘッダーで消費されないよう内部で `__` にエスケープしています（`sb_id` は `sb_id` として正しく表示されます）。

**行ごとに実行（execute_each_row）**  
前ステップの各行の値を `?` パラメータとして渡してクエリを繰り返し実行します。

```sql
-- 例: 前ステップの WP_ID をパラメータとして検索
SELECT * FROM process_log WHERE WP_ID = ?
```

**Lookup パラメータ（チェックオフ時）**  
前ステップに接続されていて SQL に `?` が含まれる場合、前ステップの**最初の行**の値を `?` に順番に当てはめて1回実行します。

**名前付きバインド `:{fieldname}`**  
前ステップ行のフィールドを名前で参照できます。DB ごとに自動変換されます。

```sql
SELECT * FROM orders WHERE customer_id = :{cust_id} AND status = :{status}
```

### File Input の設定

Excel / CSV / TSV / TXT を読み込みます。

| 項目 | 説明 |
|---|---|
| 入力フォーマット | `Excel` / `CSV` / `TSV` / `TXT` |
| ファイルパス | [参照…] で選択（または直接入力）|
| シート名 | Excel のみ。プルダウンから選択 |
| 区切り文字 | CSV/TXT: カンマ / タブ / パイプ / カスタム |
| 文字エンコーディング | `UTF-8` / `UTF-8 BOM` / `Shift-JIS`（CSV/TSV/TXT のみ）|
| 1 行目をヘッダーとして扱う | チェック ON で 1 行目をフィールド名として使用 |

**👁 プレビュー（先頭 50 行）**

ダイアログ下部の「**👁 プレビュー (先頭 50 行)**」ボタンで、現在の設定値を使って実際にファイルを読み、先頭 50 行を DataGrid に表示できます。  
内部的には `ExcelInputStep.Settings["MaxRows"] = "50"` で実行する軽量プレビューで、本番実行（MaxRows 未指定 = 無制限）と同じパース処理を通すため、エンコーディングや区切り文字の誤りもこの時点で検出できます。

### DB Output の設定

| 項目 | 説明 |
|---|---|
| DB 接続 | 接続を選択（全 6 DB 共通） |
| テーブル名 | 書き込み先テーブル |
| キーフィールド | UPSERT 時の検索キー（カンマ区切り） |
| 更新フィールド | 一致時に更新するフィールド（空 = キー以外すべて） |
| コミットサイズ | N 件ごとに中間コミット（デフォルト: 100、0 = 一括コミット） |

### File Output の設定

| 項目 | 説明 |
|---|---|
| 出力フォーマット | Excel (.xlsx) / CSV (.csv) / TSV (.tsv) / テキスト (.txt) |
| 出力ファイルパス | [参照...] ボタンで選択（または直接入力）|
| シート名 | Excel のみ設定（デフォルト: Sheet1）|
| 区切り文字 | CSV/TXT: カンマ / タブ / パイプ / カスタム |
| 文字エンコーディング | UTF-8 / UTF-8 BOM / Shift-JIS |
| ヘッダー行 | 1行目にフィールド名を出力するか |

#### ファイル名に日付・時刻を埋め込む

ファイルパスに `{フォーマット}` の形式でトークンを記述すると、実行時の日時に自動展開されます。  
ダイアログ内のボタンをクリックするとキャレット位置にトークンが挿入され、プレビューが表示されます。

| ボタン（トークン） | 展開例 |
|---|---|
| `{yyyyMMdd}` | `20260403` |
| `{yyyy-MM-dd}` | `2026-04-03` |
| `{HHmmss}` | `143022` |
| `{HH-mm-ss}` | `14-30-22` |
| `{yyyyMMdd_HHmmss}` | `20260403_143022` |
| `{yyyy}` | `2026` |
| `{MM}` | `04` |
| `{dd}` | `03` |

```
例: C:\output\report_{yyyyMMdd}.xlsx
　　→ 実行時: C:\output\report_20260403.xlsx
```

C# の標準日時フォーマット文字列（`yyyy`、`MM`、`dd`、`HH`、`mm`、`ss` など）はすべて使用可能です。

> ⚠ 出力先フォルダが存在しない場合は自動作成されます。

### Select Values の設定

フィールドの **選択・リネーム・型変換・定数挿入** を一括で行うステップです。  
ダイアログは表形式で 1 行 1 マッピングを編集します。

| 列 | 説明 |
|---|---|
| 元名 | 入力データのフィールド名（定数行では空のまま）|
| 新名 | 出力時のフィールド名（空なら元名をそのまま使う）|
| 型 | 出力時の変換型: 空 / `string` / `int` / `long` / `decimal` / `double` / `datetime` / `bool`（空 = 変換なし、パース失敗時は `null`）|
| 定数値 | 定数行の場合の固定値（元名が空のとき有効）|
| 残す | チェックを外すと出力から除外される |

**ボタン**

| ボタン | 動作 |
|---|---|
| **↓ 前段から取得** | 直前の接続元ステップを実行プレビュー（または静的解析）してフィールド名一覧を取り込む。既存マッピングと重複しない名前だけ追加。前段に DBInput / FileInput / SetVariable / SelectValues 等が必要 |
| **＋ 追加** | 空の通常行を追加 |
| **＋ 定数追加** | 元名が空で「定数値」を持つ行を追加 |
| **削除 / ↑ / ↓** | 選択行の削除・並び替え |

> 💡 「↓ 前段から取得」は、前段が SetVariable / SelectValues の場合は実行せずに設定値からフィールド名を抽出します。DBInput / FileInput の場合は **`MaxRows=5` 相当の軽い実行プレビュー**でカラム名を取得します（DBInput 系で SELECT を発行するため、対象 DB へ接続できる必要があります）。SQL に `?` / `:{name}` プレースホルダが含まれる場合も `SqlColumnExtractor` が SELECT 句から列名を抽出します。

> 💡 設定値の保存形式は 1 行 1 マッピング `元名|新名|型|残す` の `|` 区切りです。旧フォーマット（`元名=新名` / フィールド名のみ）も読み込み可能で、自動で新形式に変換されます。

**定数行の使い方**

定数のみのマッピングは、入力が 0 行でも 1 行出力します（マスタ生成用途）。

```
# 例: ConstFlag=1, ConstName="DEFAULT" の 1 行を必ず出力
元名:        新名: const_flag    型: int       定数値: 1      残す: ✓
元名:        新名: const_name    型: string    定数値: DEFAULT 残す: ✓
```

### その他のステップ設定

| ステップ | 主な設定 |
|---|---|
| DB Delete | 接続 / テーブル名 / キーフィールド / コミットサイズ |
| Insert/Update | 接続 / テーブル名 / キーフィールド / 更新フィールド / コミットサイズ |
| DB Update | Insert/Update と同じ（UPDATE のみ、INSERT しない）|
| Exec SQL | 接続 / SQL / 行ごとに実行 |
| Filter | フィールド / 演算子 / 値 |
| Calculation | 出力フィールド / 演算タイプ / フィールド 1・2 / 定数 |
| Merge Join | 結合タイプ（INNER/LEFT/RIGHT/FULL）/ キーフィールド |
| Set Variable | 変数定義（`変数名=式`、TODAY / YESTERDAY / TODAY+N など）|

> 💡 **動的な説明テキスト** — Filter / Calculation / Merge Join のダイアログでは、コンボボックスで演算子・演算タイプ・結合タイプを選ぶと、その下のグレー枠に挙動の説明文が即座に表示されます（例: `divide` を選ぶと「フィールド1 ÷ フィールド2 (0 除算は null)」）。各 ComboBoxItem にも同じ内容のツールチップが付いているので、マウスホバーでも確認できます。

## ④ ステップの接続

各ステップノードの**右端にある青い丸（出力ポート）**からドラッグして、接続先ステップへドロップします。

```
[DB Input] ──► [Select Values] ──► [File Output]
```

1. 接続元ステップの **右端の青い丸** にマウスを合わせる（カーソルが十字になる）
2. そのまま**ドラッグ**して接続先ステップへ
3. 接続先が**緑色にハイライト**されたらドロップ → 接続完成
4. キャンセルするには **Escape** キーを押す

### 接続の削除

接続線にマウスを合わせるとカーソルが手のひらに変わり、ツールチップ「右クリックで削除」が表示されます。  
**接続線を右クリック → 「この接続を削除」** を選ぶと、その接続だけが削除されます（ステップノード自体は残ります）。  
内部的には Hit-test 用に幅 12px の透明線を被せているため、矢印の細さに対して十分なクリック範囲が確保されます。

## ⑥ 編集の便利機能

キャンバスを編集する操作はほとんどショートカットでも、メニュー / 右クリックでも使えます。

### 取り消し / やり直し (Undo / Redo)

ツールバーの **↶ 取り消し** / **↷ やり直し** ボタン、または **編集メニュー**、または **Ctrl+Z** / **Ctrl+Y** で操作を巻き戻し / 復元できます。

ボタンのツールチップには次に取り消す操作の説明 (例:「取り消し: ステップ追加: DB Input (Ctrl+Z)」) が表示されます。

| 取り消せる操作 |
|---|
| ステップ追加 / 削除 (関連接続もまとめて復元) |
| 接続作成 / 削除 |
| ステップ移動 (ドラッグ / 矢印キー) |
| ステップサイズ変更 |
| 設定ダイアログでの変更 (Name + Settings 一括) |
| コピー & ペースト / 複製 (1 件にまとめて) |

新規 / 開く / クリアの直後は履歴がリセットされます。

### 複数選択

| 操作 | 動作 |
|---|---|
| 通常クリック | 他の選択を解除してそのステップ 1 つだけ選ぶ |
| **Ctrl+Click** | 選択をトグル (既存の選択を維持して追加 / 解除) |
| **Ctrl+A** | キャンバス上の全ステップを選択 |
| 空のキャンバスをクリック | 全選択解除 |

複数選択しているとき、**Delete** で一括削除、**矢印キー**で一括移動、**Ctrl+C / Ctrl+D** で一括コピー/複製ができます。

### コピー / ペースト / 複製

ステップを再利用したいときに使います。**編集メニュー**、**右クリック (ステップ上 or 空のキャンバス上)**、**ショートカット**のいずれからでも実行できます。

| ショートカット | 動作 |
|---|---|
| **Ctrl+C** | 選択中のステップと「両端が選択内に収まる接続」をクリップボードへコピー |
| **Ctrl+V** | クリップボードのステップを貼り付け (元位置から +30,+30 オフセット、Guid は新規発行) |
| **Ctrl+D** | クリップボードを介さず即座に複製 (+24,+24 オフセット) |

貼り付け / 複製は **1 件の Undo コマンドにまとまる** ので、Ctrl+Z 一回でまとめて取り消せます。

**右クリックメニューでの呼び出し**

- ステップを右クリック → 「📋 コピー」「🔁 複製」
- 空のキャンバスを右クリック → 「📋 貼り付け」「すべて選択」「🔍 検索」

クリップボードは JSON 形式なので、別のテキストエディタにペーストして内容を確認したり、別マシンの BreezeFlow に貼り付けることもできます。

### 検索 (Ctrl+F)

**Ctrl+F** または 編集メニュー → 検索 で、キャンバス左上に検索バーが現れます。

検索対象:
- ステップ名 (例: `受注取得`)
- ステップ種別 (例: `DB Input`)
- 接続設定名 (例: `本番Oracle`)
- すべての Settings 値 (SQL の中身 / FieldName / Cases / テーブル名 など)

部分一致 (大文字小文字無視)。マッチしたステップが青枠で強調されるので、そのまま **Delete / Ctrl+C / 矢印キー** に流せます。**Esc** で閉じます。

### キーボードでステップ移動

選択中のステップは矢印キーでも動かせます。

| キー | 移動量 |
|---|---|
| ←↑↓→ | 1px |
| Shift+←↑↓→ | 10px |

複数選択中は全部まとめて移動します。

## ジョブ機能

ジョブは **複数のパイプラインを指定した順番で順次実行** する機能です。  
ツールバーの **🔗 ジョブ** ボタン（または [ツール] → [ジョブ管理]）で管理ダイアログを開きます。

### ジョブの作成

1. ツールバーの「**📄 新規**」ボタンでジョブを新規作成
2. ジョブ名・コメント・ログ出力モードを設定
3. 「**＋ 追加**」でパイプラインステップを追加
4. 各ステップに **パイプラインファイルを指定**（[参照...] ボタン）
5. 必要に応じてステップのコメント・エラー時の動作を設定
6. **↑↓ ボタン** で実行順序を変更

#### ジョブ全体の項目

| 項目 | 説明 |
|---|---|
| ジョブ名 | 一意の識別名 |
| コメント | ジョブ全体への補足メモ（任意・複数行可）。`.job.json` に `Comment` として保存される |
| ログ出力モード | CLI 実行時のログファイル出力モード（後述）|

#### 各ステップの項目

| 項目 | 説明 |
|---|---|
| ファイル | 実行するパイプラインの .json ファイルパス |
| コメント | ステップに対する補足メモ（任意）。実行ログには `ファイル名 « コメント »` の形式で併記される |
| エラー時も続行する | チェックON: このステップでエラーが発生しても次のステップへ進む |

### ジョブのファイル保存・読み込み

ジョブはパイプラインと同様に **ファイル（.job.json）** として保存・共有できます。

| ボタン | 機能 |
|---|---|
| 📄 新規 | 新しいジョブを作成 |
| 📂 開く | .job.json ファイルを開く |
| 💾 保存 | 現在のジョブを既存のファイルパスに上書き保存 |
| 📋 名前を付けて保存 | 新しいファイルパスに保存 |
| ▶ 実行 | 開いているジョブをその場で実行（未保存なら保存確認） |
| ■ 停止 | 実行中のジョブをキャンセル |

> ファイルに未保存の変更がある場合、タイトルに `*` が付きます。  
> 別のジョブへ切替時またはダイアログを閉じる際に確認ダイアログが表示されます。

### ジョブの手動実行

ツールバーの「**▶ 実行**」でジョブを UI から直接実行できます。

- 未保存の変更があると「保存して実行 / 保存せず実行 / 中止」を確認します
- 実行中は他のボタンが無効化され、進捗ステータスがダイアログ下部に表示されます
- 全てのログメッセージは **メインウィンドウのログパネル** にも出力され、ダイアログを閉じた後に履歴を確認できます
- ジョブ実行中はダイアログを閉じられません（先に「■ 停止」を押す）

## スケジュール実行

ツールバーの **⏰ スケジュール** ボタン（または右パネルから）でスケジュール管理ダイアログを開きます。

### スケジュールの設定項目

| 項目 | 説明 |
|---|---|
| スケジュール名 | 識別用の名前 |
| **アプリ起動中のみ監視する** | アプリ内スケジューラで監視するかどうか（既定: OFF）。**Windows タスクスケジューラ モードでは Windows タスク側の有効/無効が実際の実行を制御** |
| **実行ファイル** | パイプライン（`.json`）またはジョブ（`.job.json`）を [参照...] で選択。種類は拡張子・内容から自動判定し「種類: パイプライン / ジョブ」と表示 |
| **実行モード** | **インアプリ** または **Windows タスクスケジューラ** |
| スケジュール種類 | 毎日 / 毎週 / 毎時 / インターバル |
| コメント | スケジュールへの補足メモ（任意・複数行可）。`schedules.json` に `Comment` として保存される |

### 一覧の見た目

スケジュール管理ダイアログの一覧では実行対象を **アイコン**で区別します。

| アイコン | 対象 |
|---|---|
| 📄 | パイプライン |
| 📦 | ジョブ |

**無効**のスケジュールは灰色文字 + 薄灰色背景でディスプレイされます。ホバーすると「(有効) / (無効)」がツールチップに表示されます。

### 実行モード

| モード | 説明 | 対応対象 |
|---|---|---|
| インアプリ | アプリが起動している間のみ監視・実行 | パイプライン・ジョブ |
| Windowsタスクスケジューラ | アプリ未起動でも OS が定期実行（管理者権限が必要な場合あり）| **パイプライン・ジョブ両対応** |

### スケジュール種別

| 種別 | 設定項目 |
|---|---|
| 毎日（指定時刻） | 実行時刻（HH:MM）|
| 毎週（指定曜日・時刻） | 曜日 + 実行時刻 |
| 毎時（指定分） | 毎時 XX 分（00〜59）|
| インターバル（N分ごと） | 実行間隔（分）|

### 右パネルのスケジュール一覧

メインウィンドウ右パネルに登録済みスケジュールが一覧表示されます。  
各カードに **📄 パイプライン / 📦 ジョブ** のアイコンが付き、種別が一目で分かります。  
各カードの **有効/無効トグル** で即座に切り替えができます（インアプリ監視のオン／オフ）。Windows タスクスケジューラ モードのスケジュールはトグルでアプリ内監視を切っても Windows タスク側で実行され続けます。

ヘッダー右の **「有効のみ」チェックボックス**（既定 ON）で表示を絞り込めます:

- **チェック ON**: 有効なスケジュールだけ表示
- **チェック OFF**: 無効も含めて全件表示

> 💡 スケジュール管理ダイアログの一覧右側の **↑/↓ ボタン** で表示順を並び替えできます。並び替えはダイアログ閉じ時にメイン画面の右パネルにも反映されます。

## 実行履歴

ツールバーの **📋 実行履歴** ボタン、または [ツール] → [実行履歴...] で実行履歴ダイアログを開きます。手動 / スケジュール / CLI / ジョブいずれの経路で動かしたパイプラインも、全件 `runs.db` (アプリと同じフォルダの SQLite ファイル) に記録されています。

### できること

- **過去 N 日の実行を一覧表示** — 状態 (成功 / 失敗 / キャンセル / 実行中)、開始時刻、所要時間、トリガ (手動 / スケジュール / CLI / ジョブ)、行数を一覧で確認
- **フィルタで絞り込み** — 期間 (DatePicker)、状態チェックボックス、トリガチェックボックス、パイプライン名 / ファイルパス LIKE 検索
- **詳細ペイン** — 1 件選ぶと右側にステップ別の実行明細 (成功/失敗、所要、行数) と当時のログを表示。失敗時のエラーメッセージもここで確認
- **📂 ファイルを開く** — 該当パイプラインの JSON をメインウィンドウで開く (未保存変更があれば確認ダイアログ)
- **🗑 削除** — 個別の履歴を削除 (関連ステップ明細・ログも一緒に消える)
- **DB 最適化** — runs.db の VACUUM を実行してファイルサイズを圧縮

### 保持ポリシー

起動時に自動で **30 日より古い実行**と **5000 件を超えた古い実行**を削除します。容量を気にせず使えますが、長期保存したい記録は別途エクスポートしてください (将来 CSV エクスポート機能を追加予定)。

### 失敗時の挙動

履歴記録はパイプライン実行を**絶対に邪魔しません**。SQLite の I/O 障害が起きても catch されてログだけ捨てられ、パイプラインそのものは普通に流れます。

## コマンドライン (CLI) モード

`BreezeFlow.exe` は GUI 起動以外に、コマンドライン引数を渡してヘッドレスで動作させることができます。Windows タスクスケジューラやバッチファイルから呼び出す場合に使用します。

| 引数 | 用途 |
|---|---|
| `--run <pipeline.json>` | 指定したパイプライン JSON を 1 回だけ実行（GUI を表示しない） |
| `--run-job <job.job.json>` | 指定したジョブを実行（複数パイプラインを順次） |
| `--convert-ktr <input.ktr> [<output.json>]` | Pentaho KTR ファイルを BreezeFlow JSON に変換 |

### 実行例

```bat
:: パイプラインをコマンドラインから実行
BreezeFlow.exe --run "C:\pipelines\daily_load.json"

:: ジョブを実行 (複数パイプラインを順次)
BreezeFlow.exe --run-job "C:\jobs\nightly.job.json"

:: KTR を変換
BreezeFlow.exe --convert-ktr "C:\old\transform.ktr" "C:\new\transform.json"
```

> 💡 **KTR 変換の警告メッセージ** — 変換時に解決できなかった項目はサマリ末尾に警告として出力されます。
> - `DBInput / ExecSQL ステップで <connection> が空` → 接続名が解決できないため `ConnectionId` 未設定で出力。生成後の JSON を開き、ダイアログで手動で接続を割り当ててください
> - `SQL 内の ? を :{p<n>} 形式にフォールバック` → 前段の Lookup ステップ名が判明する場合は「前段ステップ '<lookup>' の出力フィールドを :{p1}, :{p2} に当てはめてください」と具体的に案内します

### ログ出力

`--run` / `--run-job` で実行したログは、対象ファイルと同じフォルダに  
`<ファイル名>_<yyyyMMdd_HHmmss>.log` として保存できます。  
標準出力にも同じ内容が出力されるため、コマンドプロンプトから起動した場合はリアルタイムに進捗を確認できます。

ログファイル出力はパイプライン／ジョブごとに 3 モードから選べます（既定: エラー時のみ）。

| モード | 動作 |
|---|---|
| エラー時のみ (既定) | 例外発生時だけ `.log` を書き出す。成功時はファイルを残さない |
| 常に出力 | 成功・失敗を問わず常に `.log` を書き出す |
| 出力しない | エラーが起きても `.log` を書き出さない |

UI では、メインウィンドウ ツールバー右端の「ログ:」セレクタ（パイプライン用）と、  
ジョブ管理ダイアログの「ログ出力モード」セレクタ（ジョブ用）から切り替えます。  
設定はパイプライン／ジョブの JSON ファイルに保存されます。

### タスクスケジューラ登録時の動作

スケジュール管理ダイアログで「Windows タスクスケジューラ」モードを選んで保存すると、内部的に上記コマンドラインがタスクスケジューラに登録されます。

| Target | 登録されるコマンド |

## よくある質問

### Q. File Output でファイルが出力されない

- ファイルパスが設定されているか確認してください
- ログパネルのエラーメッセージを確認してください
- 出力先フォルダへの書き込み権限を確認してください（フォルダが存在しない場合は自動作成されます）

### Q. ファイル名に今日の日付を付けたい

File Output の設定ダイアログで `{yyyyMMdd}` トークンをファイル名に含めてください。

```
C:\output\result_{yyyyMMdd}.xlsx  →  C:\output\result_20260403.xlsx
```

ダイアログ内のボタンをクリックするとカーソル位置にトークンが挿入され、プレビューも表示されます。

### Q. 複数のパイプラインを順番に実行したい

**ジョブ機能** を使用してください。  
🔗 ジョブ ボタンからジョブ管理ダイアログを開き、実行したいパイプラインを順番に登録します。

### Q. アプリを起動しなくてもスケジュール実行したい

スケジュール管理ダイアログで **実行モード: Windows タスクスケジューラ** を選択し保存してください。  
保存時に `schtasks.exe` でタスクが登録され、アプリ未起動でも定期実行されます。  
※ パイプライン・ジョブのどちらも対応しています。  
※ 管理者権限が必要な場合があります。  
※ 既定では「ユーザーがログオンしている時のみ実行」で登録されます。未ログオン時にも動作させたい場合は、登録後に Windows タスクスケジューラ GUI で「ユーザーがログオンしているかどうかにかかわらず実行する」へ手動変更してください。

### Q. CSV / TXT を日本語で出力したい

エンコーディングを **Shift-JIS** または **UTF-8 BOM** に設定してください。  
Excel で直接開く場合は Shift-JIS が確実です。

### Q. Oracle に接続できない

DB 接続設定ダイアログの各フィールド（ホスト / ポート / サービス名 / ユーザー / パスワード）を個別に入力してください。  
内部的に `host:port/service` 形式に自動変換されます。

### Q. PostgreSQL / SQL Server / MariaDB / SQLite に接続するには

**[ツール] → [DB 接続設定]** ダイアログで「DB 種別」プルダウンから選択し、各 DB 用のフィールドを入力してください。

- **PostgreSQL**: サーバー / ポート（5432） / DB名 / ユーザー / パスワード
- **SQL Server**: サーバー / ポート（1433） / DB名 + Windows 認証チェック（または ユーザー / パスワード）
- **MariaDB**: 入力欄は MySQL と同一（ドライバを共用）
- **SQLite**: 「データベースファイル」に `.db` / `.sqlite` ファイルを指定（[参照…] ボタン）。ファイルが無ければ接続時に自動生成

### Q. SQLite ファイルが作成されない

ファイルパスのフォルダが存在しないと作成に失敗します。出力先フォルダがあるか確認してください。  
また、ファイルパスは絶対パス推奨です。相対パスはアプリの実行ディレクトリ起点になります。

### Q. コミットサイズはどう設定すればよいか

デフォルト（100件）で問題ない場合がほとんどです。  
大量データで速度を優先する場合は大きくし（500〜1000）、  
メモリを節約したい場合は小さく（10〜50）設定してください。  
`0` を指定すると全件処理後に一括コミットします。

### Q. 昨夜のスケジュール実行が成功したか確認したい

ツールバー **📋 実行履歴** から確認できます。状態 / トリガ / 期間で絞り込めるので、「失敗 + スケジュール + 過去 7 日」のような条件で深夜の異常を一覧できます。失敗した実行を選ぶと右ペインに当時のログとステップ別の失敗箇所が表示されます。

### Q. Filter で「外れた行」も別ファイルに出したい

Filter ステップの **fail ポート (オレンジ色の丸)** から別の File Output に接続してください。pass ポート (緑) と同様にドラッグで配線できます。fail を配線しなければ旧来通り不一致行は捨てられます。

### Q. 都道府県別にファイルを分けたい

**🔀 Switch** ステップを使ってください。Cases に値とポート名を 1 行ずつ書くと、その数だけ出力ポートが生えます。各ポートから個別の File Output へつなげれば自動で振り分けて書き込みされます。

### Q. 誤って消してしまった / 操作を取り消したい

**Ctrl+Z** で直前の操作を取り消せます。ステップ追加・削除・接続・移動・サイズ変更・設定変更・コピー貼り付け、すべて対象です。**Ctrl+Y** でやり直し。ツールバーの **↶ 取り消し** ボタンのツールチップに「次に何が取り消されるか」が表示されます。

### Q. 似たような DB Input をたくさん作りたい

設定済みの DB Input を選択 (青枠) してから **Ctrl+D** で複製してください。新しい Guid と +24,+24 のオフセットで真横に並びます。接続先のテーブル名だけ変えれば完成。複数同時複製も対応 (Ctrl+Click で複数選択してから Ctrl+D)。
