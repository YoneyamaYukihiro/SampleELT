# BreezeFlow 仕様書

## 概要

BreezeFlow は、ビジュアル ETL（Extract / Transform / Load）ツールです。
主要 RDBMS（**Oracle / MySQL / PostgreSQL / SQL Server / MariaDB / SQLite**）およびファイル（Excel / CSV / TXT / TSV）を対象に、データの抽出・変換・出力をドラッグ＆ドロップで設計・実行できます。
複数のパイプラインをまとめて順次実行する **ジョブ機能** と、定時・定期の **スケジュール実行機能** を備えます。

### リファクタリングの設計指針

- **DB プロバイダ抽象**: ADO.NET の差分（接続ファクトリ／プレースホルダ記法／UPSERT 構文／識別子クォート）を `DbProvider` 抽象クラスに集約し、各 DB 種別（Oracle/MySQL/PostgreSQL/SqlServer/Sqlite）は単一の Strategy として実装。新しい DB を追加する際は `DbProvider` を 1 つ実装するだけで済む。
- **DB ステップは委譲シェル**: `OracleInputStep` / `MySQLInputStep` 等は後方互換のため `StepType` 名を残しつつ、実体は `DbInputExecutor`／`DbOutputExecutor`／`DbInsertUpdateExecutor`／`DbUpdateExecutor` に委譲する 30 行未満のラッパー。
- **KTR 変換は Strategy + 登録簿**: `IKtrStepConverter` を実装した 13 種類のステップ変換器を `KtrStepConverterRegistry` で集中管理。新ステップ種別の追加は新クラス 1 ファイルのみで完結する。
- **DI 軽量版**: `Models/Stores/IConnectionStore.cs` などのインターフェイスに `Default` 静的プロパティを設け、テスト時はモックに差し替え可能。本番コードはデフォルトで Registry シングルトンを使う。
- **ダイアログサービス**: 17 種類のステップ設定ダイアログ起動ロジックを `Services/StepDialogService.cs` に集約。MainWindow から UI 起動責務を分離。
- **多ポート (分岐)**: `StepBase.OutputPorts` と `ExecuteRoutedAsync` の 2 層で拡張。ExecutionEngine は常に Routed API を呼び、単一ポートステップは既定実装で「" " ブランチに包む」だけ。Filter / Switch は両方をオーバーライドする。詳細は「出力ポートと分岐 (多ポート対応)」節。
- **Undo / Redo**: `Services/UndoManager.cs` の `IUndoCommand` スタックに、すべてのキャンバス変更を `Execute(description, doAction, undoAction)` の 3 つ組で登録する。新しい変更操作を追加する際は対応する Do/Undo ラムダを書き、`MainViewModel.UndoManager.Execute(...)` 経由で実行する。
- **実行履歴**: `Services/RunHistoryStore.cs` がシングルトン (`Instance` プロパティ) で `runs.db` を所有し、`RunHistoryProgressWriter` が `IProgress<string>` をラップして ExecutionEngine の進捗文字列を構造化記録に変換する。書き込みはすべて catch-all で本体実行を巻き込まない。詳細は「実行履歴 (Run History)」節。

## アーキテクチャ

### データフロー（パイプライン）

`StepBase` は 3 つの実行 API を持つ:

```csharp
// 多ポート対応のストリーミング API (ExecutionEngine が常に呼ぶ)。
// 出力行を BranchKey でタグ付けして返す。単一ポートステップは既定実装で良い。
IAsyncEnumerable<RoutedRow> ExecuteRoutedAsync(
    IAsyncEnumerable<Dictionary<string, object?>> input,
    IProgress<string> progress,
    CancellationToken ct)

// 単一ポートのストリーミング API (旧 API。DB Input / Output / 行単位変換はこちらをオーバーライド)。
// ExecuteRoutedAsync の既定実装はこの出力を "" タグでラップする。
IAsyncEnumerable<Dictionary<string, object?>> ExecuteStreamingAsync(
    IAsyncEnumerable<Dictionary<string, object?>> input,
    IProgress<string> progress,
    CancellationToken ct)

// レガシー List API (一括処理向け。デフォルト実装が ExecuteStreamingAsync をドレインして List 化)
Task<List<Dictionary<string, object?>>> ExecuteAsync(
    List<Dictionary<string, object?>> inputData,
    IProgress<string> progress,
    CancellationToken ct)
```

オーバーライド規則:

- **単一ポートステップ** (大多数): `ExecuteStreamingAsync` だけオーバーライドすれば良い。`ExecuteRoutedAsync` の既定実装が `""` ブランチでラップする
- **多ポートステップ** (Filter / Switch): `ExecuteRoutedAsync` をオーバーライドして行ごとに BranchKey を返す。さらに `ExecuteStreamingAsync` もオーバーライドする (レガシー `ExecuteAsync` 経由の呼び出し対策。Filter は pass のみ通過、Switch は素通し)

入出力:

- **input / inputData** : 単一前段の場合は前段の出力、複数前段の場合はその連結 (後方互換用。多入力ステップは通常 `AllInputStreams` を使う)
- **AllInputStreams** : `List<IAsyncEnumerable<...>>`。複数入力時 (MergeJoin / TableCompare 用) のみ ExecutionEngine がセット。`AllInputStreams[0]` = Left、`[1]` = Right。各ストリームは 1 回だけ列挙する責務がある
- Output ステップは入力をそのまま下流に流す (passthrough)。`DBOutputStep` は 1 行ずつ INSERT/UPSERT してから yield する

#### 出力ポートと分岐 (多ポート対応)

`StepBase.OutputPorts` は `IReadOnlyList<OutputPort>` を返す。既定は `[OutputPort.Default]`（Key="" の単一ポート）。多ポートステップはこれを上書きする。

| ステップ | 出力ポート (Key) | 意味 |
|---|---|---|
| 大多数 (DB Input / Calculation / SelectValues 等) | `""` | 既定の単一ポート |
| **Filter** | `pass` / `fail` | 一致行 / 不一致行 |
| **Switch** | Cases で動的、+ `default` | 値ごとに振り分け |

`PipelineConnection.SourceBranchKey` はこの接続が「ソースステップのどのポートから出ているか」を保持する。null/空 = 既定ポート (`""`)。

UI 表示:
- ノード右端のポート円は OutputPorts の数だけ縦に均等配置される (単一ポートのときは中央 1 個)
- 色分け: 既定=青、`pass`=緑、`fail`=オレンジ、`default`=灰
- 接続線には BranchKey ラベル (例: `pass`, `tokyo`) が出力元寄りに表示される

### 実行エンジン

`ExecutionEngine` : Kahn's Algorithm でトポロジカルソート → 依存順に IAsyncEnumerable で連鎖実行する。実行は遅延列挙で、末端ステップを engine が drain した時点で初めて上流の DB Reader まで pull が伝わる。

中間データの保持は `(Guid StepId, string BranchKey)` をキーとする 2 種類の辞書:
- `stepStream` : ストリーミングのまま下流に渡せる場合 (単ポート + 単下流)
- `stepMaterialized` : ファンアウト / 多入力下流 / 多ポート分岐の場合に List 化

メモリ戦略:
- 単入力 + 単出力の連鎖は完全ストリーミング (DB Reader → 行単位変換 → DB 書き込み / ファイル出力)。ピークメモリは「現在処理中の 1 行」レベル
- 出力ファンアウト (1 → 複数下流) は当該ステップの出力を一度 List 化して再利用可能にする
- 多入力ステップ (MergeJoin / TableCompare) は各前段ストリームを直接 `AllInputStreams` に渡し、ステップ側で必要に応じて (片側 dict 化 + もう片側ストリーミング) 消費する
- 多ポートステップ (Filter / Switch) は `ExecuteRoutedAsync` を一度ドレインし、`Dictionary<BranchKey, List<Row>>` に振り分け。Filter / Switch は行単位の判定だけなのでメモリピークの主因にならない
- 末端ステップ (DB Output / File Output) は engine が明示的にドレインして副作用 (DB 書き込み / ファイル出力) を発火させる
- 下流消費がすべて終わった中間データは `remainingConsumers` カウント (ステップ単位) で検知して全ブランチを即解放 → GC を促す
- ステップ完了 (`WrapWithProgress` / `WrapRoutedWithProgress` の `finally`) で `step.AllInputStreams` をクリアし、`ListAsAsync` が内部で捕捉していた巨大 List への参照を解放する

#### ブランチ別下流のフェッチ

下流が前段の特定ブランチを参照するときは `(predStepId, predBranchKey)` で stepStream / stepMaterialized を引く。見つからず、かつ BranchKey が空でない場合のみ既定ポート (`""`) にフォールバックする。これは「単ポートの送出元に対して空でない BranchKey を指定された場合」のセーフネットで、データを黙って捨てないために設けている。

`JobExecutor` : ジョブ内の `JobStep` を `Order` 順に並べ `ExecutionEngine` を順次呼び出す。`ContinueOnError = false` のステップでエラーが発生した場合はジョブを中断する。各パイプラインのログは `[i/N] パイプライン実行: ファイル名 « コメント »` の形式で出力される（コメント未設定時はファイル名のみ）。

### DB 接続設定

`ConnectionRegistry`（シングルトン）が全接続設定を保持し `connections.json` に永続化。各ステップは `Settings["ConnectionId"]` (Guid 文字列) で接続を参照する。

#### `DbConnectionInfo` の安全属性

| プロパティ | 既定値 | 役割 |
|---|---|---|
| `Environment` | `Development` | 環境タグ (`Development` / `Staging` / `Production`)。`EnvironmentColor` / `EnvironmentBrush` 経由で UI 色 (DEV=`#1565C0` 青 / STG=`#F57C00` オレンジ / PRD=`#D32F2F` 赤) を返し、接続マネージャの〇・ステップノードの接続ラベル色を統一する。PRD はさらにノード枠を赤太線で強調 |
| `IsReadOnly` | `false` | true のときこの接続を使う書き込み系ステップ (`DBOutput` / `DBDelete` / `DBUpdate` / `InsertUpdate` / `ExecSQL`) は `PipelineSafetyChecker` で Block 判定される。UI 表記は「Select 専用 (🔒 書き込み禁止)」 |
| `DisplayName` / `EnvironmentBadge` / `EnvironmentColor` / `EnvironmentBrush` | (計算プロパティ・全て `[JsonIgnore]`) | `DisplayName` は `🔶 [PRD] Spirytus TRN01D 🔒` のように DB アイコン + 環境バッジ + 名前 + RO アイコンを連結。`EnvironmentBadge` は `DEV` / `STG` / `PRD`、`EnvironmentColor` は色コード文字列、`EnvironmentBrush` は XAML 直接バインド用の `SolidColorBrush` を返す。`Brush` は `Dispatcher` を持つため必ず `[JsonIgnore]` を付与する (付与しないと `JsonSerializer.Serialize` が落ちて `ConnectionRegistry.Save()` が握りつぶす → 設定が永続化されない事故になる) |

#### `ResolveConnectionString` の解決ルール (誤実行防止)

```
Settings["ConnectionId"] が指定されている?
  ├ Yes
  │   ├ Guid として解釈不能 → ConnectionResolutionException 投げる
  │   ├ Registry に存在しない → ConnectionResolutionException 投げる
  │   └ 解決成功 → その接続文字列を返す
  └ No
      └ Settings["ConnectionString"] (旧形式) があればそれを返す。なければ空文字
```

ポイントは **`ConnectionId` が指定されているのに解決できない場合、旧 `ConnectionString` キーへサイレントフォールバックしない**こと。フォールバックを許すと「接続を削除したらパイプラインが古い接続文字列で別 DB を叩く」という事故が起こり得たため。

#### `PipelineSafetyChecker.Check(pipeline)`

各ステップを走査し、以下を `Issue` として返す:

| 検査 | Severity |
|---|---|
| `ConnectionId` が Guid 不正 / Registry 未登録 | **Block** (実行を完全停止) |
| Read-only 接続 (`IsReadOnly=true`) への書き込み系ステップ | **Block** |
| Production 接続への書き込み系ステップ | **Confirm** (UI なら確認ダイアログ、Engine ヘッドレスではスルー) |
| ExecSQL の SQL が SELECT のみ (`ContainsWriteSql` 偽) | 確認/拒否いずれも対象外 |

呼び出し箇所:
1. `MainViewModel.RunPipeline` — UI で Block / Confirm を MessageBox で処理
2. `ExecutionEngine.ExecuteAsync` — Block のみ。検出時は `ConnectionResolutionException` を投げ、step 単体の catch を貫通する

#### 実行ログの接続詳細

ExecutionEngine は各ステップを実行する直前に、`Settings["ConnectionId"]` を持つステップについて `ConnectionSafety.DescribeConnection(conn)` を progress に出力する。出力例:

```
[受注書込] 接続: [PRD] Spirytus TRN01D (oracle://163.141.231.40:1521/MES02)
```

これにより、ログを見れば「どの DB に書いたか」が追跡可能になる。パスワードは含まれない。

#### Oracle DataSource パーサ

`Tools/OracleDataSourceParser.Parse(dataSource, out host, out port, out service)` は、Oracle 接続文字列の `Data Source` 値を 2 種類のフォーマットからホスト / ポート / サービス名に分解する。

| フォーマット | 例 | 解釈 |
|---|---|---|
| EZ Connect | `163.141.231.40:1521/MES02` | host=`163.141.231.40`, port=`1521`, service=`MES02` |
| TNS Descriptor | `(DESCRIPTION=(ADDRESS=(PROTOCOL=TCP)(HOST=db01)(PORT=1521))(CONNECT_DATA=(SERVICE_NAME=PROD)))` | HOST / PORT / SERVICE_NAME を正規表現で抽出。SERVICE_NAME が無ければ SID も探す。引用符付き・大文字小文字混在も許容 |

`OracleConnectionStringBuilder` は保存時に EZ Connect を TNS Descriptor 形式へ正規化することがあるため、再ロード時には両方を解釈する必要がある。両形式の往復は `OracleDataSourceParserTests` の 7 ケースで検証している。

### 接続マネージャ (ConnectionManagerDialog) の編集モデル

`ConnectionManagerDialog` は明示的な保存ボタンを持たず、編集内容は次の各タイミングで自動的に `ConnectionRegistry.Save()` を呼んでファイル永続化する:

| タイミング | 実装 |
|---|---|
| 一覧の選択変更 | `ConnectionListBox_SelectionChanged` で `e.RemovedItems[0]` を `SaveEditsToModel` → `Save()`、その後 `LoadConnectionToForm` で新しい選択を表示 |
| DB 種別変更 | `DbTypeCombo_SelectionChanged` で `BuildConnectionStringFor(newType)` → `Save()` |
| 環境タグ変更 | `EnvironmentCombo_SelectionChanged` (Production への昇格は MessageBox 確認後にロールバック可能) → `Save()` |
| 🔒 Select 専用変更 | `ReadOnlyCheck_Changed` → `Save()` |
| ＋ 追加 | `IsCurrentEntryUnedited()` で「名前が『新しい接続』のままサーバーも空」を検出して連打をブロック。それ以外は新規 `DbConnectionInfo` を追加して `Save()` |
| ↑↓ 移動 / 🗑 削除 | 即 `Save()` |
| ダイアログ閉じる | `OnClosing` で最終 `SaveEditsToModel` → `Save()` |

`SaveEditsToModel` 内では同名検出 (`WarnIfDuplicateName`) を行い、同じ名前の別接続が既に存在する場合は 1 度だけ警告を出す。

参照ステップ数は `FindReferencingSteps(conn)` がコンストラクタで受け取った `Pipeline?` の `Steps` を走査して算出する。MainWindow のツールバー経由で開いた場合は `_vm.CurrentPipeline` を渡し、ステップ設定ダイアログ経由で開いた場合は null (参照表示なし)。

#### サポート DB 一覧

| 種別 | アイコン | ドライバ | 既定ポート | UPSERT 構文 | 識別子引用 |
|---|---|---|---|---|---|
| Oracle | 🔶 | Oracle.ManagedDataAccess | 1521 | `MERGE INTO ... USING DUAL` | (なし) |
| MySQL | 🐬 | MySqlConnector | 3306 | `ON DUPLICATE KEY UPDATE` | `` `name` `` |
| PostgreSQL | 🐘 | Npgsql | 5432 | `ON CONFLICT DO UPDATE` | `"name"` |
| SQL Server | 🟦 | Microsoft.Data.SqlClient | 1433 | `MERGE` | `[name]` |
| MariaDB | 🦭 | MySqlConnector（共用） | 3306 | `ON DUPLICATE KEY UPDATE` | `` `name` `` |
| SQLite | 🪶 | Microsoft.Data.Sqlite | (ファイル) | `ON CONFLICT DO UPDATE` | `"name"` |

`DbConnectionInfo.DbType` enum に基づき、`DBInputStep` / `DBOutputStep` 等が実行時に各実装へ委譲する。MariaDB は MySQL と同一のドライバを共用するため、内部的には MySQL ステップにディスパッチされる。SQLite はサーバ・ユーザー・ポートを持たず、ファイルパスのみで接続する（接続文字列例: `Data Source=C:\path\to\file.db`）。

**FieldMappingItem の各プロパティ**

| プロパティ | 説明 |
|---|---|
| SourceName | 入力フィールド名（定数行では空）|
| DestName | 出力フィールド名（空なら SourceName を継承。`EffectiveDestName` で取得）|

- IsIncluded=false の行および SourceName が空で定数値も空の行は除外
- 入力が 0 行でも、`items.All(IsConstant)` であれば 1 行を出力（マスタ生成用途）

| 前段 StepType | 戦略 |
|---|---|
| `SetVariable` | `Settings["Fields"]` を改行区切りで解析し `変数名=式` の左辺だけ抽出 |

| `DBInput` / `OracleInput` / `MySQLInput` / `PostgreSQLInput` / `SqlServerInput` / `SqliteInput` | `Settings["SQL"]` を `SqlColumnExtractor.Extract` で解析。プレースホルダ (`?` / `:{name}`) を含んでいても SELECT ... FROM 句から列名を抽出 |

### Utility グループ

#### Merge Join

| 設定項目 | 説明 |
|---|---|
| JoinType | `INNER` / `LEFT OUTER` / `RIGHT OUTER` / `FULL OUTER` |
| KeyFields | 結合キーフィールド名（カンマ区切り）|

`MergeJoinDialog` も結合タイプ選択時に `JoinTypeDescription` に説明文を動的表示する。

実装メモ: Right 側を `Dictionary<key, List<row>>` に索引化 (O(\|Right\|)) し、Left 側はストリーミングで 1 行ずつ消費する。CROSS Join (KeyFields 未指定) のみ Right を全件 List 化して総当りする。

#### Table Compare (データ移行検証用)

2 つの入力ストリーム (`AllInputStreams[0]` = Left / `AllInputStreams[1]` = Right) をキー列で突き合わせ、各比較列の値の一致／不一致を判定して出力する。`PipelineSafetyChecker` 上は書き込み系ステップではないため、Production 接続でも追加確認は出ない。

実装メモ: Right を `Dictionary<key, row>` に索引化 (O(\|Right\|)) し、Left はストリーミングで 1 行ずつ消費して MATCH / DIFF / ONLY_LEFT を判定。Left 通過後に Dict 残余を走査して ONLY_RIGHT を出す。Left が大きい移行検証ケースで効果が大きく、従来の `~\|L\|+\|R\|` メモリが `~\|R\|` に下がる。

| 設定項目 | 説明 |
|---|---|
| KeyFields | カンマ区切り。両側を突き合わせるキー (必須) |
| CompareFields | カンマ区切り。空欄なら Left/Right の全列 - キー列を自動推定 |
| NullsEqual | 両側 NULL を等しいとみなす (既定 true) |
| IgnoreCase | 文字列を大文字小文字無視で比較 (既定 false) |
| TrimStrings | 文字列を前後トリムしてから比較 (既定 false) |
| IncludeMatched | MATCH 行も出力する (既定 false。差分のみが既定) |

**出力スキーマ**

各出力行は次の列を持つ:
- キー列 (KeyFields で指定した列名そのまま)
- 各比較列 `col` について `col_L` / `col_R` (Left / Right の値)
- `_status`: `MATCH` / `DIFF` / `ONLY_LEFT` / `ONLY_RIGHT`
- `_diff_columns`: DIFF 行のみ、差異のある比較列名をカンマ区切り

**値比較ロジック (`TableCompareStep.ValuesEqual`)**

1. 両側 NULL / DBNull → `NullsEqual` の値
2. 片側のみ NULL → false
3. 両側を `decimal` に変換できれば数値比較 (`1 == 1.0`)
4. 両側 `DateTime` ならそのまま比較
5. それ以外は文字列化し、`IgnoreCase` / `TrimStrings` を適用して比較

**Right 側のキー重複**

`rightLookup` は最初の 1 件のみを採用し、`progress.Report` で「Right 側にキー重複が N 件あります」と警告する。これにより主キー違反のあるテーブル比較でも動作するが、結果は最初の出現を基準にする。

#### Switch (フィールド値で分岐)

フィールド値で行を複数の出力ポートに振り分けるステップ。Cases で「値 → BranchKey」のマッピングを宣言する。

| 設定項目 | 説明 |
|---|---|
| FieldName | 評価対象フィールド名 |
| Cases | 1 行 1 ケース。`値|BranchKey|表示名` の形式。BranchKey と表示名は省略可 (BranchKey 省略時は値そのものを Key として使用) |
| IncludeDefault | true (既定): どの Case にも一致しない行は `default` ポートへ。false: 破棄 |

**出力ポート**

`OutputPorts` は Cases から動的に生成される。例えば Cases が
```
東京|tokyo
大阪|osaka|大阪支店
名古屋
```
なら、ポートは `tokyo` / `osaka` / `名古屋` / `default` の 4 つ。

**評価ルール**

- 上から順に最初にヒットした Case の BranchKey を採用 (同一値の Case が複数あれば先勝ち)
- 比較は文字列 Ordinal 完全一致
- 評価フィールドが NULL / 欠落の場合は `default` (または破棄)
- BranchKey に `default` という名前を指定した場合、組み込みの default と衝突するため自動的に `default_case` にリネームされる

`.json` 拡張子で保存（JSONシリアライズ、PropertyNameCaseInsensitive 読み込み）。

```json
{
  "Name": "サンプルパイプライン",
  "Steps": [
    {
      "Id": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
      "Name": "DB Input",
      "StepType": "DBInput",

        "ConnectionId": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
        "SQL": "SELECT * FROM orders",
        "ExecuteEachRow": "false"
      }
    }
  ],
  "Connections": [
    { "Id": "...", "SourceStepId": "...", "TargetStepId": "...", "SourceBranchKey": "pass" }
```

`SourceBranchKey` は多ポートステップ (Filter / Switch) から出る接続でどのポートを参照するかを示す。null / 省略 = 既定の単一ポート (`""`)。

## ジョブ機能

### 概要

複数のパイプラインを指定した順序で順次実行する。ジョブ自体も `.job.json` ファイルとして保存・共有できる。

### Job モデル

| プロパティ | 説明 |
|---|---|
| Id | Guid（自動生成）|
| Name | ジョブ名 |
| Comment | ジョブ全体への補足メモ（任意・複数行可）|
| LogMode | `OnError`（既定）/ `Always` / `Never` — CLI 実行時のログファイル出力モード |
| Steps | JobStep のリスト（Order 昇順で実行）|
| FilePath | 保存先ファイルパス（JSON に含めない）|

### JobStep モデル

| プロパティ | 説明 |
|---|---|
| Order | 実行順序（0 始まり）|
| Name | コメント（任意）。`JobExecutor` の実行ログには `ファイル名 « Name »` の形式で併記される。空ならファイル名のみ表示 |
| PipelineFilePath | 実行するパイプラインの JSON ファイルパス |
| ContinueOnError | `true`：このステップでエラーが発生しても次のステップへ続行 |

### ジョブファイル形式

```json
{
  "Id": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
  "Name": "日次バッチ",
  "LogMode": "OnError",
  "Steps": [
    {
      "Id": "...",
      "Order": 0,
      "Name": "マスタ更新",
      "PipelineFilePath": "C:\\pipelines\\master_update.json",
      "ContinueOnError": false
    },
    {
      "Id": "...",
      "Order": 1,
      "Name": "集計処理",
      "PipelineFilePath": "C:\\pipelines\\aggregate.json",
      "ContinueOnError": true
    }
  ]
}
```

### ジョブのファイル管理

ジョブはレジストリを持たず、`.job.json` ファイル単体で管理される。`JobManagerDialog` で「📄 新規」「📂 開く」「💾 保存」「📋 名前を付けて保存」で操作する。「▶ 実行 / ■ 停止」で UI から手動実行も可能（ログはメインウィンドウのログパネルへ出力）。

## スケジュール機能

### ScheduleEntry モデル

| プロパティ | 説明 |
|---|---|
| Name | スケジュール名 |
| Comment | スケジュールへの補足メモ（任意・複数行可）|
| Target | `Pipeline`（パイプラインファイル直接実行）/ `Job`（ジョブファイル実行）。UI では参照ファイルの拡張子(`.job.json`)/内容(`Steps[0].PipelineFilePath`)から自動判定される |
| PipelineFilePath | Target = Pipeline の場合に使用 |
| JobFilePath | Target = Job の場合に使用 |
| Mode | `InApp`（アプリ起動中のみ）/ `TaskScheduler`（Windows タスクスケジューラ）|
| Type | `Daily` / `Weekly` / `Hourly` / `Interval` |
| IsEnabled | **InApp モード時** に `PipelineSchedulerService` が監視対象に含めるかどうか（既定: `false`）。TaskScheduler モードでは Windows タスク側の有効/無効が実際の実行を制御するため、このフラグは影響しない |

### スケジュール種別

| 種別 | 説明 | 設定項目 |
|---|---|---|
| Daily | 毎日 指定時刻 | HH:MM |
| Weekly | 毎週 指定曜日・時刻 | 曜日 + HH:MM |
| Hourly | 毎時 指定分 | MM |
| Interval | N 分ごと | 間隔（分）|

### 実行モード

| モード | 説明 |
|---|---|
| インアプリ | アプリが起動している間のみ `PipelineSchedulerService` 内の DispatcherTimer（1分周期）で監視・実行 |
| Windowsタスクスケジューラ | `schtasks.exe` でタスクを登録。アプリ未起動でも OS が定期実行。**Pipeline / Job どちらにも対応** |

スケジューラ Tick は `IScheduleStore.Default.CalcLastDueTime()` で直近の実行予定時刻を計算し、`LastRunTime` と比較して未実行であれば実行する。Target が Job の場合は `JobLoader.LoadFromFile()` → `JobExecutor.ExecuteAsync()` を呼び出す。

`LastRunTime` が `null` のエントリを初めて観測した tick では過去の予定時刻を catch-up せず、`LastRunTime = now` を記録して次の予定時刻まで待つ（スケジュール登録／有効化直後の即時実行を防ぐ）。

### Windows タスクスケジューラ登録時の挙動

`TaskSchedulerHelper.Register(entry)` が `entry.Target` で分岐し、登録するコマンドラインを切り替える。

| Target | 登録される `/TR` の中身 |
|---|---|
| Pipeline | `"<exe path>" --run "<entry.PipelineFilePath>"` |
| Job | `"<exe path>" --run-job "<entry.JobFilePath>"` |

タスクは `BreezeFlow\<サニタイズ済みスケジュール名>` という名前で登録される。タスク名で使えない文字 (`\ / : * ? " < > |`) は `_` に置換される。リネーム時は `entry.LastTaskName` に保持された旧タスクを自動削除する。

## 未保存変更の検出

### パイプライン

以下の操作で `IsModified = true` となりタイトルバーに `*` が付く。

- ステップの追加・削除・接続
- ステップ設定ダイアログを閉じた後

**保存時**（`SavePipeline`）、**新規作成**（`NewPipeline`）、**開く**（`LoadPipeline`）、**クリア**（`ClearCanvas`）の前に未保存確認ダイアログを表示。ウィンドウの `×` ボタンでも同様に確認する。

### ジョブ

以下の操作で `_isJobDirty = true` となりダイアログタイトルに `*` が付く。

- ジョブ名・コメント・ログ出力モードの変更
- ステップの追加・削除・移動（↑↓）・適用
- ステップ編集パネル下段（ファイル・コメント・エラー時続行）の編集

**ダイアログを閉じる時**に未保存確認を行う。ファイル保存（💾 保存 / 📋 名前を付けて保存）成功後にフラグをクリアする。手動実行（▶ 実行）時にも未保存なら「保存して実行 / 保存せず実行 / 中止」を確認する。

ステップ編集パネル下段の編集は `StepField_Changed` で dirty 扱いになるが、`StepsListView_SelectionChanged` での自動再読込時は `_suppressStepEvents` フラグで誤検知を防いでいる。

### ステップノードカード

- **ステップ名**（太字）・**種別ラベル**・**接続設定名**（青字）を表示
- 右端の **青い丸** が出力ポート（ドラッグして接続）
- 右下の **リサイズグリップ** でノードサイズ変更可（最小 120×50px）

### 接続線の右クリック削除

接続線レイヤー（`ConnectionsControl`）は 2 重構造になっている:

| 役割 | 表示 |
|---|---|
| Hit-test 用透明線 | `Stroke="Transparent"` / `StrokeThickness="12"` / `Cursor="Hand"` / 右クリックで `ContextMenu「この接続を削除」` を表示 |
| 表示用接続線 | `Stroke="#2196F3"` / `StrokeThickness="2"` / `IsHitTestVisible="False"` |
| 矢印 Polygon | `IsHitTestVisible="False"` |

`DeleteConnection_Click` → `MainViewModel.DeleteConnection(ConnectionViewModel)` で `Connections` コレクションと `Pipeline.Connections` を同期削除し、`AddLog` でログを出し、`IsModified = true` を立てる。両端ノードの `NotifyConnectionChanged()` を呼んでノード側の表示も更新する。

### SQL 入力 TextBox の共通キー操作

`Controls/SqlEditorBehavior.HandlePreviewKeyDown(sender, e)` を `PreviewKeyDown` イベントにフックすることで、3 つの SQL 入力ダイアログ（`SQLEditorDialog` / `ExecSQLDialog` / `DBInputDialog`）で共通のキー挙動を提供する:

| キー | 動作 |
|---|---|
| `Tab`（単押し） | 選択範囲を半角スペース `TabSize = 4` 個に置換、キャレットを直後に移動 |
| `Ctrl+Tab` | WPF 既定挙動に委譲（次のコントロールへフォーカス移動） |
| `Shift+Tab` | WPF 既定挙動に委譲（タブ文字挿入）|

各 TextBox には `AcceptsTab="True"` を併設しているため、`Ctrl+Tab` を使わない限り通常のタブ移動には進まない。Tooltip にも「Tab でスペース 4 個挿入、Ctrl+Tab で次のコントロールへフォーカス移動」を明記している。

### DataGrid ヘッダーのアンダースコア表示

WPF の DataGrid 自動列ヘッダーは `_` をアクセスキー prefix（次の文字を下線付き表示）として消費する。`ExcelInputDialog` / `DBInputDialog` のプレビューグリッドでは `AutoGeneratingColumn` イベントで `e.Column.Header` の `_` を `__` にエスケープし、`sb_id` などのカラム名を意図通り表示する。

### 右パネル（スケジュール一覧）

登録済みスケジュールを一覧表示。各カードに **📄 パイプライン / 📦 ジョブ** のアイコンと **有効/無効トグルボタン** を持ち即座に切り替え・保存できる。  
`ScheduleManagerDialog` 内で **↑/↓ ボタン**による並び替えを行うと、ダイアログ閉じ時の `RefreshSchedulePanel()` で表示順がこの右パネルにも自動反映される。

ヘッダー右の **「有効のみ」** チェックボックス（既定 ON）でフィルタする:
- ON: `entry.IsEnabled == true` のエントリのみ表示
- OFF: 全件表示

### 接続・スケジュール一覧の並び替え

`ConnectionManagerDialog` / `ScheduleManagerDialog` の一覧右側に **↑/↓ ボタン** を配置。  
- 接続側は `ObservableCollection<DbConnectionInfo>.Move()` で並び替え、`ConnectionRegistry.Save()` で `connections.json` を即時更新
- スケジュール側は `List<ScheduleEntry>` のタプル swap で並び替え、`ScheduleRegistry.Save()` で `schedules.json` を即時更新

どちらも選択状態は維持される。

## バージョン管理

`BreezeFlow.csproj` の `<Version>` プロパティに SemVer 形式（例: `0.1.0`）でバージョンを記述する。  
ビルド時にアセンブリの `AssemblyVersion` / `FileVersion` / `InformationalVersion` へ反映され、`BreezeFlow.exe` のファイル詳細プロパティにも焼き込まれる。

| 場所 | 表示形式 | 取得元 |
|---|---|---|
| ヘルプ → バージョン情報 | `Version 0.1.0` | `App.AppVersion`（`Assembly.GetName().Version?.ToString(3)`）|
| exe ファイル詳細プロパティ | `0.1.0.0` | csproj の `<Version>` |

リリース時の手順:

1. `<Version>` を SemVer に従って手動更新（バグ修正 → patch、機能追加 → minor、互換性破壊 → major）
2. コミット → プッシュ → publish

任意で GitHub Releases と連携する場合は `git tag v0.1.0 && git push origin v0.1.0` を併せて行う。

### Publish プロファイル

`Properties/PublishProfiles/win-x64.pubxml` に Windows x64 用の発行プロファイルを定義している。Visual Studio の「発行」と CLI の両方で参照される:

```bash
dotnet publish BreezeFlow.csproj -p:PublishProfile=win-x64
```

| プロパティ | 値 |
|---|---|
| Configuration | Release |
| RuntimeIdentifier | win-x64 |
| SelfContained | **true** （.NET 10 Desktop Runtime を同梱）|
| PublishDir | `publish\win-x64\` |
| PublishTrimmed | false （WPF は Trim 非対応）|

self-contained のため、配布先 PC に .NET ランタイム未インストールでも `BreezeFlow.exe` が単独で起動する。ファイル一式のサイズは約 160MB。

## エディタ UX

### 取り消し / やり直し (Undo / Redo)

`Services/UndoManager.cs` がスタックベースの取り消し履歴を管理する。`IUndoCommand { Do(); Undo(); Description }` を満たす操作を `UndoManager.Execute(description, doAction, undoAction)` で登録すると、その操作は **Ctrl+Z で逆実行 / Ctrl+Y (または Ctrl+Shift+Z) で再実行** できる。

| 操作 | 取り消し可 | 実装場所 |
|---|---|---|
| ステップ追加 | ✓ | `MainViewModel.AddStep` |
| ステップ削除 (Delete キー / コンテキストメニュー) | ✓ (関連接続もまとめて復元) | `MainViewModel.DeleteSelectedStep` |
| 接続作成 (ドラッグ) | ✓ | `MainViewModel.ConnectStepsFromPort` |
| 接続削除 (右クリック) | ✓ | `MainViewModel.DeleteConnection` |
| ステップ移動 (ドラッグ完了時) | ✓ | `MainWindow.StepNode_MouseLeftButtonUp` |
| ステップ移動 (矢印キー) | ✓ | `MainViewModel.MoveSelectedSteps` |
| ステップサイズ変更 | ✓ | `MainWindow.StepNode_MouseLeftButtonUp` |
| 設定ダイアログでの変更 | ✓ (Name + Settings 一括) | `MainWindow.OpenStepSettingsDialog` |
| コピー&ペースト / 複製 | ✓ (まとめて 1 件) | `MainViewModel.PasteFromJson` |
| 新規作成 / 開く / クリア | × (履歴リセット) | `UndoManager.Clear()` を呼ぶ |

履歴上限は 200 件 (`UndoManager(capacity: 200)`)。超えた分は最古から捨てる。`MainViewModel.CanUndo` / `CanRedo` が変化するとツールバーの「↶ 取り消し」「↷ やり直し」ボタンの enable/disable が自動更新される。ボタンの ToolTip には次に取り消す/やり直す操作の説明 (例: 「取り消し: ステップ追加: DB Input (Ctrl+Z)」) が動的に出る。

### 複数選択

`StepNodeViewModel.IsSelected` ベース。クリック規則:

- **通常クリック**: 他の選択を解除してそのステップだけ選ぶ (既存挙動)
- **Ctrl+Click**: 選択をトグル (既存選択を維持して追加 / 解除)
- **空キャンバスクリック**: 全選択解除

`MainViewModel.GetSelectedSteps()` で IsSelected=true なステップ一覧を取得する。`SelectedStep` は「最後にクリックされた主選択」(設定ダイアログを開く対象など) として残し、複数選択の集合は `IsSelected` 経由で表現する。

### コピー / ペースト / 複製

| ショートカット | 動作 |
|---|---|
| `Ctrl+C` | 選択中の全ステップと**両端が選択内に収まる接続**を JSON にしてクリップボードへ |
| `Ctrl+V` | クリップボードの JSON を読み、Guid を振り直して `(+30, +30)` 位置オフセットで貼り付け |
| `Ctrl+D` | クリップボードを介さず即座に複製 (`+24, +24` オフセット) |
| `Ctrl+A` | すべてのステップを選択 |

クリップボード形式は `BreezeFlowSteps` カスタム形式と Unicode テキストの両方を書き込む。後者があるため、別のテキストエディタで内容を確認 / 修正 / 別マシンへ送ることもできる。

貼り付け / 複製は **1 件の Undo コマンドにまとめる** — Ctrl+Z 一回で複数ステップと接続を一括で消せる。

### コピペ機能の発見性 (アクセス経路)

ショートカット以外にも以下の経路から同じ操作にアクセスできる。GUI のどこからでも見えるようにしてある:

| 経路 | 項目 |
|---|---|
| **編集メニュー** (上部 「編集(E)」) | 取り消し / やり直し / コピー / 貼り付け / 複製 / 削除 / すべて選択 / 選択解除 / 検索 |
| **ステップ右クリック** | 設定 / コピー / 複製 / 削除 |
| **キャンバス空白部分の右クリック** | 貼り付け / すべて選択 / 選択解除 / 検索 |
| **ツールバー** | ↶ 取り消し / ↷ やり直し ボタン |
| **キーボード** | Ctrl+Z/Y/C/V/D/A/F、Delete、矢印 (±Shift) |

ステップを右クリックでコピー/複製した時、その時点でクリックされたステップが未選択なら自動的に単独選択にしてから操作する (一般的な GUI の挙動に合わせる)。

### キャンバス検索 (Ctrl+F)

キャンバス左上にオーバーレイの検索バー (`FindBar`) が現れる。検索対象:

- ステップ名 (`Step.Name`)
- 種別ラベル (`TypeLabel`)
- 接続設定名 (`ConnectionLabel`)
- すべての `Settings` 値 (SQL / FieldName / Cases / 接続文字列ヒント など)

すべて `IndexOf(..., OrdinalIgnoreCase)` での部分一致。マッチしたステップは `IsSelected = true` で青枠強調されるので、見つけた直後に Delete / コピー / 矢印移動などにそのまま流せる。Esc または ✕ ボタンでバーを閉じる。

### キーボードでステップ移動

選択中のステップを矢印キーで動かせる:

| キー | 移動量 |
|---|---|
| ←↑↓→ | 1px |
| Shift+←↑↓→ | 10px |

複数選択中はまとめて移動。1 回のキー押下が 1 件の Undo コマンド (連続押しても 1 件ずつ積まれるので、Ctrl+Z で 1 ステップずつ戻る)。

## 実行履歴 (Run History)

パイプライン / ジョブの実行を SQLite (`runs.db`) に永続化し、UI から検索・詳細表示・履歴からのパイプライン再オープンができる。

### スキーマ (`runs.db`)

```sql
CREATE TABLE runs (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    run_guid      TEXT    NOT NULL,
    pipeline_path TEXT,
    pipeline_name TEXT,
    job_path      TEXT,
    job_name      TEXT,
    trigger       TEXT    NOT NULL,    -- manual / schedule / cli / job
    started_at    TEXT    NOT NULL,
    ended_at      TEXT,
    duration_ms   INTEGER,
    status        TEXT    NOT NULL,    -- Running / Success / Failed / Cancelled
    error_message TEXT,
    total_rows    INTEGER
);

CREATE TABLE run_steps (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id        INTEGER NOT NULL REFERENCES runs(id) ON DELETE CASCADE,
    step_order    INTEGER NOT NULL,
    step_name     TEXT    NOT NULL,
    step_type     TEXT    NOT NULL,
    started_at    TEXT    NOT NULL,
    ended_at      TEXT,
    duration_ms   INTEGER,
    row_count     INTEGER,
    status        TEXT    NOT NULL,
    error_message TEXT
);

CREATE TABLE run_logs (
    id        INTEGER PRIMARY KEY AUTOINCREMENT,
    run_id    INTEGER NOT NULL REFERENCES runs(id) ON DELETE CASCADE,
    timestamp TEXT    NOT NULL,
    level     TEXT    NOT NULL,        -- INFO / WARN / ERROR
    message   TEXT    NOT NULL
);
```

`PRAGMA journal_mode=WAL; PRAGMA synchronous=NORMAL; PRAGMA foreign_keys=ON;` を `EnsureSchema()` のたびに適用。`ON DELETE CASCADE` で run 削除時にステップ / ログも自動消去。

### 記録の単位

- **パイプライン単体実行** (`MainViewModel.RunPipeline` / `App.RunHeadlessAsync` / `PipelineSchedulerService` Pipeline 経路): 1 run = 1 パイプライン
- **ジョブ実行** (`JobExecutor.ExecuteAsync`): ジョブ内の **各パイプラインごとに 1 run** が記録される。`job_name` / `job_path` が denormalize されて入り、フィルタで「このジョブの履歴」を一覧できる
- **trigger 値の決め方**:
  - UI の ▶ 実行 → `manual`
  - スケジュール → `schedule`
  - `--run` / `--run-job` CLI → `cli`
  - `JobExecutor` の既定 → `job` (UI / CLI / スケジュール いずれかから明示的に渡されない場合)

### Progress 統合 (`RunHistoryProgressWriter`)

`ExecutionEngine` の API は無変更。`IProgress<string>` をラップする `RunHistoryProgressWriter` が、ExecutionEngine が出す固定文字列パターンを正規表現で検出して `run_steps` に変換する:

| パターン | 動作 |
|---|---|
| `[Name] 実行中...` | `RecordStepStart(runId, ++order, Name, "")` |
| `[Name] 完了 (N行)` | `RecordStepEnd(stepId, Success, N, null)` |
| `[Name] エラー: msg` | `RecordStepEnd(stepId, Failed, null, msg)` |
| `[Name] キャンセル…` | `RecordStepEnd(stepId, Cancelled, null, "キャンセル")` |
| その他 | `run_logs` に INFO/WARN/ERROR (エラー: 検出 / 警告: 検出 で分類) として記録 |

ラッパーは元 progress に常にパススルーするので、UI ログパネルや Console 出力には影響を与えない。

末端の `[Name] 完了 (N行)` は `LastReportedRowCount` として記録され、`runs.total_rows` 候補として `EndRun` に渡される。

未完了ステップ (パイプライン中断時の取りこぼし) は `FinishUnclosedSteps(status, errorMessage)` でまとめてクローズする。各統合ポイントの catch 節で呼ぶ。

### 保持ポリシー

`App.OnStartup` で `RunHistoryStore.ApplyRetention(RetentionDays, MaxRows)` を呼ぶ。既定値は **30 日 / 5000 件**。両条件を OR で評価し、どちらかに該当する行を削除する。`runs.db` 自体のサイズ圧縮は UI の「DB 最適化」ボタン (VACUUM) から手動で実行できる。

### UI (`RunHistoryDialog`)

ツールバー「📋 実行履歴」/ ツールメニュー「実行履歴...」から開く。

| 領域 | 内容 |
|---|---|
| 上部フィルタ | 期間 (DatePicker) / パイプライン名 LIKE / 状態チェック (成功/失敗/キャンセル/実行中) / トリガチェック (手動/スケジュール/CLI/ジョブ) |
| 左ペイン | run 一覧 DataGrid (状態アイコン / 開始時刻 / 所要 / トリガ / パイプライン名 / 行数) |
| 右ペイン上 | 選択中 run のヘッダ (ファイルパス / ジョブ名 / トリガ / 行数 / エラー) |
| 右ペイン中 | run_steps 明細 DataGrid (状態 / # / ステップ名 / 所要 / 行数) |
| 右ペイン下 | run_logs を時系列で表示 (ERROR は橙、WARN は黄で着色) |
| 下部 | 📂 ファイルを開く (該当パイプラインをメインウィンドウで読み込む) / 🗑 削除 / DB 最適化 / 閉じる |

「📂 ファイルを開く」を押すと `DialogResult=true` で閉じ、`MainWindow` 側で `LoadPipelineFromFile(SelectedPipelinePath)` を呼ぶ。未保存変更がある場合は確認ダイアログを挟む。

### 失敗時の振る舞い

`RunHistoryStore` のすべての書き込み API は内部で `try { ... } catch { }` し、SQLite I/O 例外がパイプライン本体を巻き込まないようにしている。`Instance` が `null` (初期化失敗時) でも、各統合ポイントは `if (Instance != null)` で記録をスキップするだけで、実行自体は通常通り続行する。
