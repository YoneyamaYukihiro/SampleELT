# BreezeFlow テスト仕様書

## 概要

`BreezeFlow.Tests` プロジェクトは xUnit ベースのユニットテスト群で、本番コード `BreezeFlow.csproj` の挙動を検証する。
合計 **217 件 / 25 クラス**。

| 観点 | 内容 |
|---|---|
| フレームワーク | xUnit 2.9.0 (`Microsoft.NET.Test.Sdk` 17.12.0) |
| ターゲット | `net10.0-windows7.0` (WPF 参照のため Windows 限定) |
| 主な依存 | `Microsoft.Data.Sqlite` (DB ステップの E2E 検証用)、`EPPlus` (Excel 出力検証用) |
| 並列実行 | xUnit 既定で並列。静的シングルトン (`IConnectionStore.Default`) を差し替えるテストは `[Collection("ConnectionStore")]` で直列化 |

## 実行方法

```bash
# 全件実行
dotnet test BreezeFlow.Tests/BreezeFlow.Tests.csproj

# クラス絞り込み
dotnet test BreezeFlow.Tests/BreezeFlow.Tests.csproj \
  --filter "FullyQualifiedName~PipelineSafetyCheckerTests"

# テスト名絞り込み
dotnet test BreezeFlow.Tests/BreezeFlow.Tests.csproj \
  --filter "FullyQualifiedName~Production"
```

## カバレッジ計測

```bash
# Coverlet (xUnit と同梱の coverlet.collector で動く) でカバレッジ取得
dotnet test BreezeFlow.Tests/BreezeFlow.Tests.csproj \
  --collect:"XPlat Code Coverage" \
  --results-directory ./TestResults

# ReportGenerator で HTML レポートを生成 (要 dotnet tool)
dotnet tool install -g dotnet-reportgenerator-globaltool   # 初回のみ
reportgenerator \
  -reports:"TestResults/**/coverage.cobertura.xml" \
  -targetdir:"TestResults/CoverageReport" \
  -reporttypes:Html
```

生成された `TestResults/CoverageReport/index.html` を開くと、行・ブランチ・メソッド別カバレッジが見られる。

> 💡 `coverlet.collector` パッケージは現状参照されていない。本格的にカバレッジを取りたい場合は `BreezeFlow.Tests.csproj` に `<PackageReference Include="coverlet.collector" Version="6.0.x" />` を追加すること。

## 追加を推奨するテスト

優先度の高い順。事故の起こりやすさ・回帰の見逃しやすさで判断。

### 🔴 高優先度

| 提案 | 理由 |
|---|---|
| **`SelectValuesStep` の `FieldMappingItem.Parse` / `Serialize` ラウンドトリップ** | 新フォーマット `元名\|新名\|型\|残す` への移行で、旧フォーマット (`=` 区切り / 名前のみ) との後方互換が壊れると既存パイプラインで実行時に失敗する |
| **`SelectValuesStep.ConvertValue` の型変換** | int/long/decimal/double/datetime/bool それぞれの正常系 + パース失敗時 null フォールバック。データ精度バグの早期発見 |
| **`ConnectionSafety.ContainsWriteSql` の DML/DDL 判定** | `PipelineSafetyChecker` の安全機構の中核。コメント内キーワード、複数文、CTE 内 UPDATE などのエッジケース回帰防止 |
| **`SqlColumnExtractor.Extract` のパース** | SELECT 句に `DISTINCT` / `TOP N` / 関数 / alias / コメント / プレースホルダ込みのバリアント。Select Values の「↓ 前段から取得」が依存 |
| **`FilterStep` の `greaterOrEqual` / `lessOrEqual`** | 本番に実装済みだが現状テスト無し。仕様書では存在するため回帰を検知できない |
| **`PipelineLoader` で Switch ステップを含む JSON の往復** | Cases / IncludeDefault が SourceBranchKey と共に正しく保存・復元されるか。現状は Filter のマイグレーションのみテスト済み |

### 🟠 中優先度

| 提案 | 理由 |
|---|---|
| **`MergeJoinStep.RightOuterJoin`** | 4 種類のうち 1 種類のみテスト不足 |
| **`PipelineSafetyChecker` の追加ケース**: 同一接続を使う複数の書き込みステップ / 複数の Block と Confirm の混在 | 現実のパイプラインに近いシナリオ |
| **`ConnectionRegistry.Save`/`Load` のラウンドトリップ** (`Environment` / `IsReadOnly` 追加後の互換性) | 旧 connections.json から新フィールド既定値で読み込めるか |
| **`PipelineLoader` の不正 JSON / 欠損キー耐性** | エンドユーザーがファイルを手編集する可能性あり |
| **`UndoManager` 統合シナリオ**: AddStep → Connect → Move → Undo Undo Undo → Redo Redo Redo が完全に元に戻るか | 個別 API のテストはあるが「操作チェーンが対称に巻き戻る」全体検証は無い |
| **`RunHistoryStore` の並列書き込み** (10 スレッド × 100 件同時 BeginRun + AppendLog) | WAL モードのスレッドセーフ性を担保。実運用でスケジュール + 手動が同時実行されたとき重要 |
| **`SwitchStep` の `IncludeDefault=true` で default が暗黙ポートとして生える境界** | Cases に "default" という名前を書いた場合の自動リネーム (`default_case`) など |

### 🟡 低優先度 (時間が取れるとき)

| 提案 | 理由 |
|---|---|
| **`ExcelInputStep` のフォーマット切替** (Excel / CSV / TSV / TXT × エンコーディング 3 種) | 実ファイルでの検証。サンプル `.xlsx` / `.csv` を `BreezeFlow.Tests/TestData/` に置く |
| **`KtrToJsonConverter` の追加ステップ種別** | 現在 3 件のみ。13 種ある Strategy のうち未網羅 |
| **`PipelineSchedulerService.SchedulerTick` の catch-up 抑制ロジック** | スケジュール登録/有効化直後の即時実行を防ぐ部分のテスト |
| **`OracleDataSourceParser` の不正入力** (途中で閉じカッコが欠落、複数 HOST/PORT、IPv6 アドレス) | パーサの堅牢性 |
| **`RunHistoryStore.Vacuum`** | 書き込み中の VACUUM が安全か (現状 catch-all なので即座に致命的にはならないが) |

## メンテナンス

- 新規テストを追加したら **本仕様書も更新** すること (`dotnet test` の合計件数も)
- テストクラスを新設する場合は本仕様書の「テスト一覧」に新セクションを追加
- 静的シングルトン (`IConnectionStore.Default` 等) を差し替えるテストは `[Collection("<名前>")]` を付け、並列実行で衝突しないようにする
